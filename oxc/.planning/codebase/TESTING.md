# Testing Patterns

**Analysis Date:** 2026-04-08

## Test Framework

**Runner:**
- Built-in Rust test harness: `cargo test`
- Snapshot testing framework: `insta` v1.42.1 with YAML format
- Config: `insta.toml` or inline configuration (feature: `["yaml"]`)

**Assertion Library:**
- Insta snapshots for output validation
- Standard Rust assertions: `assert_eq!()`, `assert!()` for specific checks
- No external assertion library (assert macros are native Rust)

**Run Commands:**
```bash
cargo test                    # Run all tests
cargo test -p qwik-optimizer # Run optimizer crate tests only
cargo test <test_name>        # Run specific test
cargo insta review            # Review and approve snapshot changes
cargo insta accept            # Accept all snapshot changes
cargo test -- --nocapture    # Show println output during tests
```

## Test File Organization

**Location:**
- Test input files: `optimizer/src/test_input/` — separate directory from source
- Snapshots: `optimizer/src/snapshots/` — organized by test module path
- Unit tests: inline `#[cfg(test)] mod tests { }` within source files

**Naming:**
- Test input files: match test function name exactly
  - `test_example_1` reads from `test_example_1.tsx`
  - `test_example_capture_imports` reads from `test_example_capture_imports.js`
  - Multi-file tests use directories: `test_project_1/` directory
- Snapshot files: auto-generated by insta, follow pattern:
  - `qwik_optimizer__js_lib_interface__tests__example_1.snap`
  - `qwik_optimizer__transform__tests__test_example_1_...snap`

**Structure:**
```
optimizer/
├── src/
│   ├── test_input/
│   │   ├── test_example_1.tsx
│   │   ├── test_example_2.tsx
│   │   ├── test_example_capture_imports.js
│   │   ├── test_example_jsx.js
│   │   ├── test_example_ts.tsx
│   │   └── test_project_1/
│   │       └── src/
│   │           └── ...multiple input files
│   └── snapshots/
│       ├── qwik_optimizer__js_lib_interface__tests__example_1.snap
│       ├── qwik_optimizer__js_lib_interface__tests__example_2.snap
│       └── ...
├── js_lib_interface.rs   (contains test module)
├── illegal_code.rs       (contains test module)
└── segment.rs            (contains test module)
```

## Test Structure

**Suite Organization:**

All snapshot tests follow the macro-driven pattern. Example from `js_lib_interface.rs`:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use glob::glob;
    use serde_json::to_string_pretty;
    use std::path::PathBuf;

    #[test]
    fn test_example_1() {
        assert_valid_transform_debug!(EntryStrategy::Segment);
    }

    #[test]
    fn test_example_2() {
        assert_valid_transform!(EntryStrategy::Segment);
    }
}
```

**Patterns:**
- Setup: none required (test input files are pre-created in `test_input/`)
- Teardown: none (snapshots are managed by insta)
- Assertion: macros handle snapshot comparison automatically

**Commented/Disabled Tests:**
- Disabled with `// #[test]` comment before function
- Examples in `js_lib_interface.rs`: `test_example_9`, `test_example_10`
- Comments explain why: `// Not removing:` or `// Not converting:`

## Test Macros

**Main Testing Macros:**

Located in `macros.rs`. These macros auto-discover test input files and manage snapshots.

**`assert_valid_transform!($entry_strategy)`:**
```rust
#[test]
fn test_example_2() {
    assert_valid_transform!(EntryStrategy::Segment);
}
```
- Automatically loads `test_example_2.tsx` or `test_example_2.js`
- Runs transform with specified entry strategy
- Captures result in snapshot
- Assertion happens via `insta::assert_snapshot!()`

**`assert_valid_transform_debug!($entry_strategy)`:**
```rust
#[test]
fn test_example_1() {
    assert_valid_transform_debug!(EntryStrategy::Segment);
}
```
- Same as above but also prints `println!("{:?}", result)` during test
- Useful for development to see detailed output
- Both `debug` and non-debug versions produce identical snapshots

**`assert_processing_errors!(|errors| { /* verify errors */ })`:**
```rust
assert_processing_errors!(|errors: Vec<ProcessingFailure>| {
    assert_eq!(errors.len(), 2);
    if let ProcessingFailure::IllegalCode(IllegalCodeType::Function(_, Some(name))) =
        &errors[0]
    {
        assert_eq!(name, "hola");
    }
});
```
- Runs transform and collects processing failures
- Passes errors to closure for custom verification
- Used for error-case testing (currently commented out in examples)

**`snapshot_res!($res, $prefix)`:**
```rust
snapshot_res!(result, format!("==INPUT==\n\n{}", code.to_string()))
```
- Internal macro handling snapshot format
- Outputs module-by-module results with metadata
- Includes diagnostics array

**`function_name!()` macro:**
```rust
let func_name = function_name!();
```
- Extracts current function name at compile time
- Used to match test input files: `test_example_1` → `test_example_1.tsx`
- Located in `macros.rs`

## Mocking

**Framework:** None — insta snapshots replace mocking

**Patterns:**
- No mocks used in transform tests
- Instead: use real test input files with complete code samples
- Snapshots capture full output including intermediate structures

**What to Mock:**
- Nothing — snapshots provide better visibility than mocks

**What NOT to Mock:**
- AST parsing (always uses real parser)
- Transform logic (tests run full pipeline)
- Code generation (tests verify actual output)

## Fixtures and Factories

**Test Data:**

No explicit fixture libraries. Test data is static files:

```typescript
// test_input/test_example_1.tsx
import {$, component, onRender} from '@builder.io/qwik';

export const renderHeader = $(() => {
    return (
        <div onClick={$((ctx) => console.log(ctx))}/>

```

**Location:**
- Individual files: `optimizer/src/test_input/test_example_*.{tsx,js}`
- Multi-file projects: `optimizer/src/test_input/test_project_1/` with subdirectories
- No factory pattern; files are manually created

**Test Projects:**
- `test_project_1/` contains multiple input files (represented as directory)
- Used for integration tests spanning multiple modules
- Loaded with custom logic (not using standard macro)

Example from `js_lib_interface.rs`:
```rust
#[test]
fn test_project_1() {
    let func_name = function_name!();
    let path = PathBuf::from("./src/test_input").join(func_name);
    // Custom handling for multi-file test

```

## Test Types

**Unit Tests:**

Located within source files in `#[cfg(test)]` modules:

- `illegal_code.rs`: Tests `is_illegal_code_in_qrl()` trait implementations
  ```rust
  #[test]
  fn test_function_declaration_is_illegal() {
      let allocator = Allocator::default();
      let stmt = parse_statement("function foo() {}", &allocator);
      let result = stmt.is_illegal_code_in_qrl();
      if let Some(IllegalCodeType::Function(_, name)) = result {
          assert_eq!(name, Some("foo".to_string()));
      }
  }
  ```
- `source.rs`: Tests `Source::from_file()` functionality
- `segment.rs`: Tests segment naming logic

Scope: Individual function/trait behavior without full pipeline

**Integration Tests:**

Via snapshot tests in `js_lib_interface.rs`:

- `test_example_*`: Single-file transformation tests with various code patterns
- `test_project_1`: Multi-file project transformation
- Scope: Full transform pipeline from parsing to code generation
- Validates output structure, extracted components, diagnostics

**E2E Tests:**

No dedicated E2E framework. Snapshots serve as E2E validation:

- Each test captures complete transform output
- Snapshots include: transformed code, extracted components, diagnostics
- Re-running `cargo test` re-validates against snapshots
- `cargo insta review` shows diffs if behavior changes

## Common Patterns

**Async Testing:**

Not used — all tests are synchronous.
- Transform is CPU-bound, runs on current thread
- NAPI binding handles async/tokio (not tested here)

**Error Testing:**

Two approaches used:

1. Snapshot-based (primary):
   ```rust
   #[test]
   fn test_example_capturing_fn_class() {
       assert_valid_transform_debug!(EntryStrategy::Segment);
   }
   ```
   Expected output includes any errors in snapshot

2. Assertion-based (alternative, currently disabled):
   ```rust
   assert_processing_errors!(|errors: Vec<ProcessingFailure>| {
       assert_eq!(errors.len(), 2);
       match &errors[0] {
           ProcessingFailure::IllegalCode(...) => {}
           _ => panic!("Expected error type")
       }
   });
   ```

**Snapshot Format:**

Snapshots in YAML-like format containing:

```
==INPUT==
[original source code]

============================= [output file 1 path] (ENTRY POINT)==
[transformed code]
None
/*
[SegmentAnalysis JSON]
*/

============================= [output file 2 path] ==
[transformed code]

== DIAGNOSTICS ==
[diagnostic array]
```

**Updating Snapshots:**

When transform behavior changes intentionally:
```bash
cargo test                # Run tests, snapshots fail
cargo insta review        # Interactive review of changes
# Accept changes, snapshots update
```

## Test Conventions

**Naming:**
- Descriptive test function names: `test_example_1`, `test_example_jsx`, `test_project_1`
- Pattern: `test_` prefix followed by scenario name
- Disabled tests: `// #[test]` prefix with reason comment

**Input File Organization:**
- Minimal but complete code examples
- Mix of TypeScript (`.tsx`), JavaScript (`.js`)
- Various patterns tested: JSX, imports, handlers, classes, functions

**Snapshot Verification:**
- Regenerate snapshots only after intentional code changes
- Review diffs carefully: check both code and diagnostics
- Commit `.snap` files to git (treat as golden references)
